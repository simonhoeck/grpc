Setup-Anleitung

Server-Installation auf einem neuen Rechner

Diese Anleitung führt dich Schritt für Schritt durch die komplette Installation des gRPC-Datenerfassungssystems. Am Ende empfängt der Server alle 7 BESI-Services und speichert die Daten vollständig in MongoDB.

🐳
Docker Desktop
Redis, MongoDB & Grafana laufen als Container
🐍
Python 3.10+
Für generate_stubs.py (nur bei Proto-Änderungen)
🏭
Port 50051
gRPC-Eingang für alle BESI-Services
Schritt 1

1
Docker Desktop installieren
Für Redis, MongoDB und Grafana

Lade Docker Desktop von docker.com/products/docker-desktop herunter und installiere es. Nach der Installation einmal neu starten.

Prüfen ob Docker läuft:

docker --version
Erwartete Ausgabe
✓ Docker version 27.x.x, build ...

Git installieren
Zum Klonen des Repositories

Lade Git von git-scm.com herunter und installiere es.

git --version

Für Server und Worker

Schritt 2

Repository klonen

2
Quellcode herunterladen
Einmalig auf dem Server-PC ausführen

Öffne eine Eingabeaufforderung oder PowerShell und wähle einen geeigneten Ablageort (z. B. C:\Apps):

git clone https://github.com/simonhoeck/grpc.git
cd grpc

Das Verzeichnis enthält nach dem Klonen u. a.:

  • server.py — gRPC-Server, empfängt alle 7 BESI-Services
  • worker.py — liest Redis, schreibt vollständig in MongoDB
  • generated/ — kompilierte Python-Stubs (44 Dateien)
  • docker-compose.yml — Redis, MongoDB, Grafana
Schritt 3

.env-Datei erstellen

3
Zugangsdaten konfigurieren
Im Projektordner eine Datei namens .env anlegen

Erstelle im geklonten Ordner (grpc/) eine Datei .env mit folgendem Inhalt:

dotenv 
# .env — diese Datei NICHT ins Git committen
MONGO_USER=admin
MONGO_PASSWORD=SicheresPasswort123
GRAFANA_USER=admin
GRAFANA_PASSWORD=SicheresPasswort123
⚠️
Passwörter ändern! Die Standard-Passwörter (changeme) niemals in einem Netzwerk verwenden. Wähle starke, einzigartige Passwörter — vor allem wenn der Rechner von außen erreichbar ist.
Schritt 4

Alle Services starten

4
Alle 6 Services als Docker-Container starten
Redis, MongoDB, Grafana, Mongo Express, gRPC-Server und Worker

Im Projektordner (grpc/) ausführen:

docker compose up -d

Dieser Befehl startet alle 6 Container — darunter auch den gRPC-Server und den Worker. Kein manuelles python server.py oder python worker.py notwendig. Docker lädt die Images automatisch herunter (nur beim ersten Mal). Danach Status prüfen:

docker compose ps
Alle Container müssen „healthy" oder „Up" zeigen
✓ grpc_redis — Status: healthy
✓ grpc_mongodb — Status: healthy
✓ grpc_mongo_express — Status: Up
✓ grpc_grafana — Status: Up
✓ grpc_server — Status: Up
✓ grpc_worker — Status: Up

Verbindung einzeln testen

Redis:

docker exec grpc_redis redis-cli ping

Antwort: PONG

MongoDB:

docker exec grpc_mongodb mongosh -u admin -p SicheresPasswort123 --authenticationDatabase admin --eval "db.adminCommand('ping')"

Antwort: { ok: 1 }

ℹ️
MongoDB braucht beim ersten Start ca. 30–60 Sekunden bis der Health-Check grün wird. Wenn docker compose ps noch „starting" zeigt, kurz warten.
Betrieb

Autostart-Verhalten

✓
Automatischer Neustart nach Crash oder PC-Neustart
Konfiguriert über restart: unless-stopped in docker-compose.yml

Alle 6 Docker-Services sind so konfiguriert, dass sie sich automatisch neu starten — ohne dass jemand eingreifen muss.

EreignisVerhalten
Container-Absturz oder FehlerAutomatischer Neustart
PC-Neustart / Docker-Daemon-NeustartAlle Services starten automatisch
docker compose downServices gestoppt — kein Autostart bis zum nächsten up -d
docker compose stop workerNur dieser Service gestoppt, kein Autostart für ihn
ℹ️
Es wird kein externer Dienst-Manager (Windows Task-Planer, systemd o. Ä.) benötigt. Docker selbst übernimmt den Autostart.

Autostart deaktivieren

Falls alle Services dauerhaft gestoppt werden sollen (z. B. für Wartung):

docker compose down

Nur den Worker stoppen (Server und Infrastruktur laufen weiter):

docker compose stop worker

Autostart wieder aktivieren

Nach einem manuellen Stopp alle Services wieder starten und den Autostart reaktivieren:

docker compose up -d

Schritt 6

Diesen Befehl in einer Eingabeaufforderung mit Administratorrechten ausführen (Rechtsklick → „Als Administrator ausführen").

Schritt 8

Alles verifizieren

8
Datenfluss von der Anlage bis MongoDB prüfen
Sobald die Anlage verbunden ist und Berichte sendet

Server-Log beobachten

In den Docker-Logs des Containers erscheinen Einträge wie:

docker compose logs -f grpc_server
2025-03-23 10:15:32 [server] alignment  component=a1b2c3d4-...  production=prod-001
2025-03-23 10:15:32 [server] bonding    component=a1b2c3d4-...  production=prod-001
2025-03-23 10:15:33 [server] pbi        component=a1b2c3d4-...  production=prod-001

Redis-Stream prüfen

docker exec grpc_redis redis-cli xlen data_stream

Gibt die Anzahl der empfangenen Nachrichten aus. Im Normalbetrieb sollte dieser Wert nahe 0 bleiben (Worker verarbeitet schnell).

docker exec grpc_redis redis-cli xlen data_stream_failed

Muss 0 sein — alles andere bedeutet Schreibfehler in MongoDB.

MongoDB-Daten prüfen

docker exec grpc_mongodb mongosh -u admin -p SicheresPasswort123 --authenticationDatabase admin grpcdata --eval "db.alignment_reports.countDocuments({})"

Gibt die Anzahl gespeicherter Alignment-Berichte aus. Wiederhole für die anderen Collections:

db.alignment_reports.countDocuments({})
db.bonding_reports.countDocuments({})
db.creation_reports.countDocuments({})
db.pbi_reports.countDocuments({})
db.pickup_reports.countDocuments({})
db.transfer_reports.countDocuments({})
db.preproduction_reports.countDocuments({})

Daten im Browser ansehen

Öffne http://localhost:8081 (Mongo Express). Dort unter der Datenbank grpcdata alle 7 Collections mit den vollständigen Dokumenten durchsuchen.

✅
Wenn Dokumente in den Collections sichtbar sind und der Failed-Stream leer bleibt, ist die Installation vollständig erfolgreich.
Referenz

Die 7 gRPC-Services

Der Server implementiert jeden dieser Services mit einer unary PushReport-Methode. Die Anlage ruft den jeweiligen Service nach jedem Prozessschritt auf.

Service Proto-Package MongoDB Collection Beschreibung
AlignmentDataService besi.component.services.alignment.v1_0_0 alignment_reports Uplooking-Camera-Adjust, Bonding-Position-Adjust, Theta-Iterationen
BondingDataService besi.component.services.bonding.v1_0_0 bonding_reports Bond-Profil (Stage-Positionen) und optionale Traces aller Achsen
CreationDataService besi.component.services.creation.v1_0_0 creation_reports Komponenten-Erstellung: Tool-Name, Lot-Info, Quell-Material
PostBondInspectionDataService besi.component.services.pbi.v1_0_0 pbi_reports Post-Bond-Inspektion: XY-Offset und Theta-Offset nach dem Bonden
PickupDataService besi.component.services.pickup.v1_0_0 pickup_reports Pickup-Prozess: Flipper-Z/Theta, Eject-Needle, Vakuum-Traces
TransferDataService besi.component.services.transfer.v1_0_0 transfer_reports Transfer vom Flipper zum Bonding-Head inkl. Component-Center-Inspektion
PreproductionDataService besi.production.services.preproduction.v1_0_0 preproduction_reports Maschinen-Name, Seriennummer, PPID, verwendete Komponenten-Typen
Referenz

MongoDB Collections

Jedes Dokument in allen Collections hat die gleiche Grundstruktur:

json 
{
  "_id":           ObjectId("..."),
  "received_at":   ISODate("2025-03-23T10:15:32Z"),   // UTC-Zeitstempel des Empfangs
  "service":       "alignment",                        // Service-Name
  "component_id":  "a1b2c3d4-e5f6-...",               // aus report.id.id
  "production_id": "prod-2025-001",                    // aus report.productionId.id
  "report": {                                          // vollständiges Proto-Objekt
    "id": { "id": "a1b2c3d4-e5f6-..." },
    "alignments": { "alignments": [ ... ] },
    "productionId": { "id": "prod-2025-001" }
  }
}

Indexes sind auf received_at, component_id und production_id angelegt — Abfragen über alle drei Felder sind performant.

Dokumente einer Komponente über alle Prozessschritte verknüpfen:

mongosh 
// Alle Reports zu einer component_id
db.alignment_reports.find({ component_id: "a1b2c3d4-..." })
db.bonding_reports.find({ component_id: "a1b2c3d4-..." })
db.pbi_reports.find({ component_id: "a1b2c3d4-..." })

// Alle Reports einer Produktion über alle Collections
["alignment","bonding","creation","pbi","pickup","transfer"].forEach(s => {
  print(s + ": " + db[s + "_reports"].countDocuments({ production_id: "prod-2025-001" }))
})

Betrieb

Datensicherung & Wiederherstellung

💾
Welche Daten müssen gesichert werden?
Drei Docker-Volumes enthalten persistente Daten
VolumeInhaltPriorität
mongo_dataAlle 7 Collections mit MaschinendatenHoch — täglich sichern
grafana_dataGrafana-Dashboards und KonfigurationMittel — nach Dashboard-Änderungen
redis_dataRedis AOF/RDB-Persistenz (Puffer)Niedrig — optional, da Daten bereits in MongoDB
⚠️
Backup-Verzeichnisse nicht im Projektordner ablegen — docker compose down -v löscht alle Volumes dauerhaft. Backups auf einem separaten Laufwerk oder Netzlaufwerk speichern.
↑
MongoDB sichern
Empfohlen täglich als geplante Aufgabe

Datenbankdump im Container erstellen:

bash 
docker exec grpc_mongodb mongodump \
  -u admin -p SicheresPasswort123 \
  --authenticationDatabase admin \
  --db grpcdata \
  --out /tmp/backup

Dump auf den Host kopieren (Datum wird automatisch eingefügt):

bash 
docker cp grpc_mongodb:/tmp/backup ./backup/mongodb_$(date +%Y%m%d_%H%M%S)
↑
Grafana sichern
Nach Änderungen an Dashboards ausführen
bash 
docker run --rm \
  -v grpc_grafana_data:/data \
  -v $(pwd)/backup:/backup \
  alpine tar czf /backup/grafana_$(date +%Y%m%d_%H%M%S).tar.gz -C /data .
↓
MongoDB wiederherstellen
Backup-Ordner-Name anpassen (z. B. mongodb_20250323_103000)

Backup in den Container kopieren:

bash 
docker cp ./backup/mongodb_DATUM grpc_mongodb:/tmp/restore

Datenbank wiederherstellen:

bash 
docker exec grpc_mongodb mongorestore \
  -u admin -p SicheresPasswort123 \
  --authenticationDatabase admin \
  --db grpcdata \
  /tmp/restore/grpcdata
⚠️
Bei einem mongorestore werden vorhandene Dokumente mit gleicher _id überschrieben. Wenn eine vollständige Rücksetzung gewünscht ist, Collection vorher leeren oder --drop anhängen.
↓
Grafana wiederherstellen
Dateiname anpassen (z. B. grafana_20250323_103000.tar.gz)
bash 
docker compose stop grafana

docker run --rm \
  -v grpc_grafana_data:/data \
  -v $(pwd)/backup:/backup \
  alpine tar xzf /backup/grafana_DATUM.tar.gz -C /data

docker compose up -d grafana
